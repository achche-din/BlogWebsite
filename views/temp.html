What is Functional Dependency?

 

In relational database theory, a functional dependency is a constraint between two sets of attributes in a relation from a database. Given a relation R, a set of attributes X in R is said to functionally determine another set of attributes Y, also in R, (written X → Y) if, and only if, each X value in R is associated with precisely one Y value in R; R is then said to satisfy the functional dependency X → Y. lets look at an example

EmployeeID EmployeeName DepartmentID DepartmentName
1 Ashutosh 1 R&D
2 Sachin 2 Software Engineering
3 Meghaj 1 R&D

Here we have a functional dependency, Department ID → DepartmentName what it implies is that suppose we have two DepartmentID deptid1 and deptid2 then if it is the case that deptid1 = deptid2 then it is guaranteed that their corresponding DepartmentName deptname1 and deptname2 should also be same. Other than this we also have other functional dependencies here (Employee ID → Employee Name, Employee ID → Department ID)

1st Normal Form(1NF)

 

  • Rules for First Normal form. A table must satisfy all the rules specified below to be in First Normal Form.

    • Each column values must be atomic. Columns cannot contain sub-columns in the first normal form. That is, you cannot list multiple cities in one column and separate them with a semi-colon. “Chicago” is atomic; whereas “Chicago; Los Angeles; New York” is not. 

    • Table should not contain repeating groups of columns. Multiple fields in a single table should not be used to store similar data. For example, column names like Telephoneno1, Telephoneno2, Telephone3 in a table all store similar data but use multiple columns. What if a customer gets a new telephoneno then we have to change schema of the table to add column telephoneno4 to the table. Clearly it is not a good design.

    • Table records must be uniquely identified using a primary key.

Second Normal Form(2NF)

 

  • Rules for Second Normal form. A table must satisfy all the rules specified below to be in Second Normal Form.

    • It is in first normal form.

    • A non-prime attribute of a relation is an attribute that is not a part of any candidate key of the relation. Table should not have any non-prime attribute that is functionally dependent on any proper subset of any candidate key of the relation(also called partial dependency). Since proper subset can also be empty so all non-prime attributes which donot depend on candidate key of table also violates 2NF.

  • Once a table is in second normal form, we are guaranteed that every column is dependent on the primary key. When all the columns relate to the primary key, they naturally share a common purpose, such as describing an employee. Every Column of the table should serve to describe what the primary key of the table identifies, if it is not the case then that column must be moved to another table.

Table-1 candidate key: {CustomerId,TelephoneId}.

{FirstName, SecondName} are non-prime attributes and they depend on proper subset{CustomerId} of candidate key hence 2NF is violated.

Third Normal Forms(3NF)

 

  • A transitive dependency is a functional dependency in which X -> Z (X determines Z) indirectly, by virtue of X -> Y and Y -> Z (where it is not the case that Y -> X). Ex. Greater than comparison(>) is transitive since if A is greater than B, and B is greater than C, then it follows that A is greater than C.

  • Rules for Third Normal form. A table must satisfy all the rules specified below to be in Third Normal Form.

    • A table is in 2nd normal form.

    • It contains only columns that are non-transitively dependent on the primary key that is all the attributes (e.g. database columns) are functionally dependent on solely the primary key

Let's look at an example to understand it better.

CustomerId CustomerName CustomerCity CustomerPostalCode
1 Pushpendra  Mathura 281006
2 Sachin Pune 411015
3 Meghaj Haryana 122011

Table-3

Table-3 is in 2NF. With 2NF we have all attributes depend on primary key but there could be dependencies between columns that could cause inconsistency. There occurs transitive Dependency: {CustomerCity -> CustomerPostalCode} in Table-3 as CustomerId -> CustomerCity while CustomerCity is not functionally dependent on CustomerId and CustomerCity -> CustomerPostalCode. This transitive dependency can cause update anomaly as you could update the CustomerPostalCode without making a corresponding update to the CustomerCity. Hence this transitive dependency needs to be removed.

Let's normalise Table-3 to 3NF 

We divide Table-3 into two tables Table-3.1 and Table-3.2 according to the rule below:

Let R be the schema that is not in 3NF due to the transitive dependency (X->Y) then we replace R in our design with (XUY) and (R- (Y-X)). Here X={CustomerCity}, Y={CustomerPostalCode}, R={CustomerId,CustomerName,CustomerCity,CustomerPostalCode}.
